PPA Payment Components — Consumer Requirement Specification1. 

Executive Summary for ConsumersConsumers require a low-effort integration to implement standard Citi payment forms.

 The provided solution consists of two scoped npm packages that handle 8 specific form sections, including Debtor, Creditor, and Intermediary bank details.
  These components are designed to be "Pure Presenters," meaning they handle the "Look and Feel" while the consumer handles the "Action". 
   Core Consumer BenefitsReduced Development Time: No need to build complex ISO 20022 forms from scratch.  
   Consistency: Standardized steel blue-gray styling and navy gradient headers across all consuming apps.  
   Built-in Validation: 12 mandatory fields and real-time hardcap checks are handled automatically within the component.  2. Functional Requirements (The Consumer Experience)2.1 The Maker Journey (<pm-maker-form>)The consumer integrates the Maker form as the first step of the payment lifecycle.Hardcap Validation: Consumers require real-time feedback on transaction amounts. The component debounces keystrokes (400ms) and expects the parent to provide a HardcapCheckResponse.  Submission Control: The consumer's host application remains the "source of truth." The submit button lives in the parent app and is enabled only when the component emits isFormValid = true.  API Orchestration: When the user clicks "Submit," the component emits the full Pain001Model payload, which the consumer then POSTs to their specific makerSubmitUrl.  2.2 The Checker Journey (<pc-checker-form>)The consumer uses the Checker component for the secondary approval step.Read-Only Review: By default, all maker-submitted fields are pre-populated and disabled (cursor: not-allowed).  Controlled Visibility: The component renders nothing until the consumer sets [visible]="true", at which point it automatically triggers a GET call to fetch the maker data.  Dual Boundary Check: If the checker modifies the Transaction Amount, the consumer expects the component to fire an amountMismatchUrl POST before the final approval.  3. Technical Requirements & IntegrationConsumers must meet the following technical baseline to use the packages:3.1 Environment BaselineFramework: Angular 17, 18, or 19.  Styling: Bootstrap 5.3+ must be present in the host's angular.json.  Providers: provideHttpClient() must be configured in the host's main.ts.  3.2 Registry AccessConsumers access the packages via the npm-icg-teamdev-local registry. Integration requires a one-time configuration of the .npmrc file with a valid NPM_AUTH_TOKEN.  3.3 Field CustomizationConsumers have the requirement to hide or relabel fields based on their specific region or module. This is achieved via the fieldConfig input, allowing consumers to map DEFAULT_FIELD_CONFIG to set hidden: true or custom labels.  4. Quick Troubleshooting for ConsumersCORS/Access: 403 errors during install usually mean the CI/CD pipeline lacks read access to the scoped registry.  Environment: If the form doesn't appear, ensure provideHttpClient() is in the main.ts providers list.  Testing: Consumers can use the built-in useMockApi: true to test the UI flow before their backend endpoints are fully ready.
